## 2026-07-11 — The assistant's command catalog is complete again, and more toolbar wiring moves to event delegation

### Fixed
- **Loading a project with non-canonical device IDs silently broke its LAGs.** On load, `_migrateState` canonicalises device IDs that don't already match the `<type-prefix><n>` scheme (e.g. an imported/generated `core1` switch becomes `sw1`) and remaps every reference — links, port keys, and LAG group keys — to the new IDs. But the port-side LAG reference (`ports[].lagGroup`) was only remapped for the SNMP-discovered `snmp-lag-…` format, while the LAG-group *map keys* were remapped for **every** format. So a project using the `lag-<deviceId>-poN` naming (the demo/import format) ended up with the map key renamed to `lag-sw1-po1` but the port still pointing at `lag-core1-po1` — every LAG left with a dangling reference and an orphan group, i.e. LAG membership silently lost. Both sides now go through one shared `remapLagId` helper, so the port reference and the map key are remapped identically and stay aligned (idempotent across repeated loads). App-created projects were never affected (their IDs are already canonical and manual LAGs use an ID-free `lg-<uid>` key). Surfaced by a full smoke test of the 500-device enterprise project. `src/app.js` (+1 real-Chrome regression test).
- **The AI assistant's on-screen command catalog was silently losing entries as the interface migrated off inline handlers.** The assistant answers "how do I do X in InfraNet" by grounding on the *real* toolbar buttons — their labels and tooltips, derived from `netmapper.html` (`lib/ui-catalog.js`) rather than the manual, so it never invents a command. That extraction read a button's action **only from its `onclick` attribute**, so every button rewired to the new `data-act` event-delegation scheme quietly dropped out of the catalogue (Save, the Report menu, rack management and others had already gone). `extractCatalog` now falls back to `data-act` as the citable action, so a delegated button is catalogued exactly like a classic one — the catalogue is back to its full size and the assistant cites the current buttons again. `lib/ui-catalog.js` (+1 unit test).

### Changed
- **The static header menus and project toolbar buttons now dispatch through event delegation instead of inline `onclick` — an internal refactor with no change in behaviour.** As part of retiring the frontend's transitional `window` bridge (see [Architecture §10](ARCHITECTURE.md)), the account menu, the Report menu and the New / Rename / Duplicate / Delete / Save toolbar buttons were rewired: the HTML carries a `data-act="…"` marker and a single delegated listener (`src/app-delegation.js`) routes the click to the imported module function, so those functions no longer need to be published on `window`. Each menu's items are registered by the module that owns them (the menu owner registers the toggle; `saveProject` is imported for the `Ctrl+S` shortcut). Every button behaves identically — verified live and by the real-Chrome E2E suite. `src/app-auth.js`, `src/app-core.js`, `src/app.js`, `src/app-audit.js`, `src/app-spare.js`, `src/app-l3.js`, `src/app-wifi.js`, `netmapper.html`.
- **The AI assistant's toolbar/panel buttons moved to event delegation, and the delegation harness now also handles `change` and `input` events — not only clicks.** The six assistant controls (open, configure, clear, send, save-config, preview-config) were rewired to `data-act`, dropping the functions from `window` while the shortcut/entry path keeps `openAssistant`. The harness (`src/app-delegation.js`) was then generalised beyond clicks: it keeps one delegated listener per event type on the document — `data-act` for click, `data-change` for `change` (selects, checkboxes, file inputs, committed numbers), `data-input` for live `input` (typing) — so form controls can leave inline `onchange`/`oninput` behind too. The first two non-click surfaces are migrated as proof: the rack-size number field (`change`) and the palette search box (`input`). Behaviour is identical — verified live and by the real-Chrome E2E suite (a delegated `change`/`input` event drives the real function). `src/app-delegation.js`, `src/app-ai.js`, `src/app-search-zoom-rack.js`, `netmapper.html`.
- **The CSV and DHCP import dialogs' change/input controls moved to event delegation too.** The file pickers, the live-lease vendor selector and the paste-area live previews (five `onchange`/`oninput` handlers across the two import modals) now carry `data-change`/`data-input` markers and their functions (`loadCsvFile`, `previewCsv`, `loadDhcpFile`, `previewDhcp`, `updateDhcpVendorFields`) are off `window`. No change in behaviour — the E2E suite drives a real delegated `input` event on each paste-area and confirms the preview renders. `src/app-csv-import.js`, `src/app-dhcp-import.js`, `netmapper.html`.

## 2026-07-09 — Discover names Windows PCs, and Stealth mode randomizes its scan order

### Added
- **Windows PCs finally get a name in Discover.** A Windows host speaks no SNMP and rarely advertises its computer name over mDNS, so it used to fall back to just its IP. Discovery now resolves the name from **NetBIOS** — a single **NBSTAT** (NetBIOS node-status) query sent **directly over UDP 137** per *alive, still-nameless* host — so a `DESKTOP-JH2S14O` shows by name in ~40 ms. (The `nbtstat` CLI queries the target on *every* local interface and waits out the dead ones, so on a host with many virtual NICs — VMware, Bluetooth — it takes 10–30 s+ and the name is lost; the direct UDP query avoids that entirely and is cross-platform. The CLI stays as a Windows-only fallback.) It runs on the **Normal and Safe** cadences (a single NBSTAT on a host you've already pinged is legitimate name resolution, within Safe's "light anti-IDS", with a gentler concurrency on Safe) and is **off on Stealth** — the true anti-IDS mode that must carry no extra NetBIOS footprint. The machine running InfraNet shows up in its own scan, but `nbtstat` cannot query its own IP ("host not found"), so the **local host is named from `os.hostname()`** — in every cadence, no network probe. Windows-only (nbtstat); a host with NetBT disabled but SMB open still relies on the existing SMB signal. `server/routes/discovery.js`, `lib/i18n.js`.

### Changed
- **Stealth mode now randomizes the scan order — it was still sweeping IPs in sequence.** The Stealth cadence already serialised the sweep and jittered the timing (a fixed interval is a detectable cadence), but it still probed `.1, .2, .3 …` in strict order — and a **sequential IP sweep is a scan signature just like a fixed interval**. Stealth now **shuffles the target order** (Fisher-Yates) on **both** phases that put packets on the wire — the base ping sweep **and** the SNMP/enrichment phase (each an independent shuffle) — and paces the enrichment phase with the same jittered delay, so neither carries a sequential pattern. New pure `_shuffled` (injectable rand, unit-tested) in `server/netscan.js`. Verified end-to-end against the real `/api/discover` route (ping + SNMP order sequential on Normal/Safe vs shuffled on Stealth; concurrency 64/32/1 and 16/8/1; NBSTAT on Normal+Safe only, local host named in every cadence). Normal and Safe are unchanged in ordering. `server/routes/discovery.js`.

## 2026-07-09 — The Verifica report stops repeating itself, and the asset register lists assets (not the cabling)

### Changed
- **"Non verificabili" is no longer a separate finding — it is folded into the "Reti del progetto" section, under the subnet that could not be reached.** The Verifica report used to show two things that told the same story: a per-network *Project networks* section (each `/24` flagged *covered / blocked / open*) and a separate device-level *Non verificabili* bucket (documented devices whose subnet the sweep never reached). They were two projections of one fact — *"this segment is off-limits from here"* — so the report said it twice. Now the network is the single container: after a Verifica each `/24` carries a **presence badge** (green *presence verified* when the sweep observed it, grey *not reached* when it did not), and the non-verifiable devices are **nested under their own subnet** as a drill-down that keeps their ignore / investigate / explain actions. The un-verifiable count still leads the banner, so nothing is lost from the summary; a non-verifiable device whose `/24` has no network row (rare) falls back to a small tail group so it can never disappear. The two axes stay distinct on the row — *topology reconstructable?* (covered/blocked/open) vs *presence confirmed?* (the badge) — because an off-segment subnet can be reachable for topology yet blind for presence. New pure join `annotateNetworksVerification` in `lib/project-networks.js` (fully unit-tested); `lib/drift-report.js` now exposes `sweepRan` + `observedSubnets`; render in `src/app-drift.js`, `lib/i18n.js` (it+en), `styles/09-user-theme.css`.
- **The PDF asset register lists IT assets, not the structural cabling.** The *Registro asset* (an ISO 27001 A.5.9 inventory) was including wall ports and electrical panels — building infrastructure with no network identity (no IP, MAC, brand or model), so they added empty rows. They are now excluded by a vendor-neutral class (`isFloor && isPassive && !hasIP` — wall sockets + electrical panels), and the "N devices documented" count follows the filtered list. Patch panels, blank panels and cable managers stay (they are rack equipment with a location); UPS / PDU / ATS stay (they have an IP). The REST API v1 inventory and the floor plan are unchanged — a wall port is still part of the topology, just not an *asset*. `lib/api-shape.js` (`isStructuralCabling`), `server/routes/export.js`.
- **Discover dialog and Contesto progetto panel — UI consistency.** The **Cadenza** (scan-pacing) dropdown in Discover now matches the theme of the fields around it — dark fill, thin hairline border, softly-rounded corners like the Subnet field — instead of the browser default, at the same size. The **Contesto progetto** panel title is shown in uppercase, consistent with the section headers below it. Cosmetic only; the DOM text is unchanged (screen readers and tests still read the normal-case string). `styles/07-modals.css`, `src/app-properties.js`, `src/app-properties-floor.js`, `styles/04-floor-rack.css`.

## 2026-07-08 — Closed-port devices finally identify themselves: mDNS + SSDP + ONVIF discovery

### Added
- **An opt-in mDNS/SSDP listen pass makes the "silent" devices name themselves — phones, tablets, smart TVs and smart appliances that answer nothing on TCP or SNMP.** A washing machine, a Chromecast, or a phone with a randomized MAC used to fall back to a low-confidence `pc` guess: they expose no management port, no web banner, no SNMP, so the classifier had only the MAC vendor to go on. But those same devices are chatty on multicast — they announce their services via **mDNS** (Bonjour / DNS-SD) and **SSDP** (UPnP). The new **"mDNS/SSDP listen"** toggle in the Discover dialog sends one mDNS query and one SSDP M-SEARCH on the local segment, listens briefly, and reads the advertised service back as a MEASURED, vendor-neutral device type — the same idea as RTSP→camera or the existing Google Cast probe, generalized to every consumer family:
  - `_googlecast` / `_airplay` / UPnP `MediaRenderer` / Roku / DIAL / Fire TV → **tv**; `_ipp` / `_printer` → **printer**; `_hap` (HomeKit) / `_matter` / Hue / ESPHome → **iot**; `_apple-mobdev2` → **mobile**; UPnP `InternetGatewayDevice` → **router**; `MediaServer` → **nas**.
  - The announced **model and manufacturer** (mDNS TXT records + the UPnP description) feed brand/model recognition, so a Chromecast, a "Samsung QN90A" or an "LG ThinQ Washer" is *named*, not just typed. The friendly mDNS model (Cast `md`) is preferred over an internal UPnP codename (e.g. a Chromecast reporting "Eureka Dongle"), and the mDNS query sets the **QU (unicast-response) bit** (RFC 6762 §5.4) so replies are received even when UDP 5353 is already held by another mDNS responder — common on Windows (Bonjour), where the multicast socket otherwise gets nothing.
  - A device that ignores ping but announces via multicast is now **discovered** (marked present) even when the ping sweep missed it.
  - **IP cameras / NVRs via ONVIF WS-Discovery.** The same sweep also sends an ONVIF **WS-Discovery** Probe (SOAP-over-UDP multicast on `239.255.255.250:3702`, the industry-standard camera discovery). A genuine ONVIF camera answers with a ProbeMatch whose scopes carry its **name** and **hardware** model, and confirm the device is a `webcam`. A ProbeMatch is only trusted when it carries a real `onvif://` scope — Windows PCs and printers also answer WS-Discovery (WSD / Function Discovery), and they must not be mistaken for cameras. Bonjour `_device-info` icon placeholders (`model=Xserve`/`RackMac`, which NAS/Samba report for the Finder server icon) are treated as generic, not a real model, so the device keeps its actual hostname.
  - **Exact model, where the device exposes it.** For ONVIF cameras the sweep follows the ProbeMatch up with an ONVIF **GetDeviceInformation** call (best-effort, unauthenticated) and prefers the commercial `Model` ("RLC-810A") over the internal hardware code the scope carries ("IPC-122"). For SNMP devices the lightweight discovery probe now also reads the standard **ENTITY-MIB** `entPhysicalModelName` of the chassis, so a switch/router/NAS that implements it reports its exact model ("GS1900-8", "C9300-24T", "DS918+") — one extra GET, vendor-neutral, silently skipped when the device has no ENTITY-MIB.
- **Readable device names in Discover.** The *Name* column (and the name of an imported node) now shows the device's own identifier instead of an opaque mDNS UUID like `2e10a349-…-….local`. Since each row already carries Vendor and Type in their own columns, the Name is kept to just that — the **hostname** when meaningful, else the announced **model** cleaned of the vendor prefix and any type descriptor ("SHIELD Android TV" → "Shield", "HUAWEI Mediapad M5" → "Mediapad M5"), else the **IP** as a stable per-device reference. No device type is appended to the name. Only the non-personal product model is kept client-side; a UPnP `friendlyName` (which can be a person's name) seeds the hostname once, cleaned, and is never re-exposed.
- **Honest scoping and safety.** Multicast is link-local (mDNS TTL 1), so the sweep only sees devices on the **same subnet** — cross-subnet would need an mDNS reflector, and the feature says so rather than pretending. The announced type is a *candidate* weighted below a real SNMP/banner signal (vendor ≠ type still holds); confidence stays manual-first. It is **opt-in** (off by default, respects the anti-IDS cadence). The UPnP description fetch is constrained to the responding device's own IP (no SSRF to arbitrary hosts), and the DNS/XML parsers are bounded (no compression-loop, entity-expansion or oversize abuse). New pure module `lib/discovery-mdns.js` (query build + wire-format parse + service→type map, fully unit-tested); the socket layer `_mdnsSsdpSweep` in `server/netscan.js`; an additive vote in `engine/fusion-scorer.js`; the opt-in `mdns` flag in `/api/discover`.

## 2026-07-07 — One classifier of record: the client stops re-typing, one shared regex table, and 40+ more vendors recognized by SNMP

### Changed
- **Discovery now has a single authoritative classifier, its pattern tables live in one shared place, and the sysObjectID coverage was widened with an industry dataset — so a TV or printer behind a switch is no longer mis-typed, and dozens of plugin-less vendors are recognized.** Three linked changes (surveyed the field first — Fingerbank, netdisco/SNMP::Info, nmap, IETF/IoT standards, Cisco ISE/ClearPass — to avoid reinventing; verdict: no open, offline, network-signal-based classification DB is redistributable, so we keep our engine and adopt only what is license-clean):
  - **The client defers to the server class whenever the server produced one.** The Discover UI kept its own thin `_guessType`/`_discSanitizeDeviceClass` heuristic and, for a *low-confidence* row, re-classified on top of the server — which is how a TV or printer sitting **behind a switch named "GS1900"** could be relabeled `switch` (the generic `gs\d{3,4}` model pattern was tested *before* the TV/printer patterns, and matched the neighbour's name). The server fusion engine always returns a class using every measured signal, so the client now **trusts it whenever present**; its own regexes run only as a genuine last-resort fallback for a row the server couldn't type at all. **Manual-first overrides (a device already in the project, a saved type hint) still win over everything.** The ordering bug is also fixed (specific TV/printer/NAS signals before the generic switch/`gs\d` check), in both the fallback and the server.
  - **One shared regex table.** The ~28 vendor/model type-patterns lived in three hand-synced copies (the fusion scorer, the legacy parity classifier, and the client). They now come from a single `lib/device-patterns.js`, consumed by the authoritative fusion scorer and the client fallback, so the two can't drift again. A new `tests/classify-golden.test.js` (55 representative devices across every type, real + lab + cross-vendor) freezes the classifier's output and **proves the extraction changed nothing**.
  - **40+ more vendors recognized from their sysObjectID.** The canonical OID→type table (`lib/device-signatures.js`) gained the enterprise-number → role facts from **netdisco/SNMP::Info** (BSD-3-Clause — the only surveyed dataset with a compatible license and offline use). This covers single-purpose vendors that had **no dedicated plugin**: firewalls (Check Point, Stormshield, NetScreen, Genua, pfSense/OPNsense), a WLC (Trapeze), SD-WAN (Riverbed Steelhead, Silver Peak), APs (Aerohive, Cambium), switches (Extreme, Enterasys, Force10, H3C, Aruba CX, Pica8, Cumulus, Ciena…) and routers (VyOS, Nokia Timetra, Juniper ERX…). Enterprise-level, so weighted **below** a model banner — a measured signal still overrides it (vendor ≠ type). `sdwan` and `wlanctrl` join the classifier's priority; a new WLAN-controller detector (vendor-neutral: "Cisco Controller"/AireOS/AIR-CT/Catalyst 9800/Aruba mobility controller) types a WLC as `wlanctrl` instead of a switch; and the classifier's type names are cross-walked to the de-facto **Nmap** device-type vocabulary (names only — internal keys unchanged so saved projects are safe).
- Validated by the golden + the legacy↔fusion parity net + a headless corpus; every previously-correct device unchanged. `src/app-discovery.js`, `src/app-discovery-classify.js`, `engine/fusion-scorer.js`, `lib/device-patterns.js` (new), `lib/device-signatures.js`, `netmapper.html`, `tsconfig.json` (+ golden + G8 SNMP::Info coverage tests).

## 2026-07-07 — The MAC vendor stops deciding the device type: switches, phones/tablets and Cast media now classify correctly

### Changed
- **Four vendor-neutral rules, driven by a live scan of a real mixed home LAN, close the last classification gaps — with zero per-vendor code.** The failures were concrete: a Zyxel-branded managed switch whose web page literally reads *"Intelligent Switch"* was typed **router**; an NVIDIA Shield and a Chromecast came out **pc**/**iot**; a Huawei tablet and phones generally landed on **pc** at a misleadingly high 92%. The root was always the same anti-pattern — a signal derived from the **MAC vendor's identity** (the OUI plugin's device-type default) outranking a **measured** signal. The fix is the method, applied generally:
  - **(1) The OUI device-type is demoted to the identity tier.** A per-vendor OUI plugin proposes a *candidate* type (Zyxel→router, D-Link→router, …), but that guess was scored high enough (up to 80) to beat a real banner/model/port/SNMP signal. It's now weighted like the other identity hints (~45), so **any** measured signal wins — one rule for every OUI plugin, mono- and multi-product vendors alike, so a Zyxel switch (which the vendor also makes) is read from its *"Intelligent Switch"* banner, not from the fact that Zyxel *also* makes routers. No vendor plugin was edited. Fixes the whole Zyxel/D-Link/Netgear/TP-Link→router class at once.
  - **(2) Google Cast is detected as a protocol, not a brand.** A new probe reads the Cast device-info endpoint (`/setup/eureka_info` on :8008) and the deep scan learns the two Cast control ports (8008/8009) — exactly the pattern already used for RTSP→camera and IPP→printer. Any Cast device of any make (Chromecast, Nvidia Shield, Nest, cast-enabled TVs) → a media endpoint (`tv`). One consistent policy: every Cast signal → `tv`.
  - **(3) Phones and tablets classify as `mobile`, decided by the OS not the brand.** The OS fingerprint already recognised Android/iOS (and tagged them "mobile") but emitted `pc`; it now emits the existing `mobile` device type. Android-TV and Cast are still separated out above it (→ `tv`); a Mac stays `pc`.
  - **(4) Confidence is honest when nothing was measured.** When a device is known only by a vendor/OS inference (its MAC + an OUI/fingerprint guess) with no measured behavioural signal — no SNMP, banner, open port, NetBIOS, SMB share or Cast — the confidence is capped, so a MAC-only guess is never presented as near-certain (manual-first: the user confirms). A device with any measured signal is uncapped as before.
- **Three more, surfaced by a live scan of the multivendor lab (`10.10.99.0/24`):**
  - **(5) `sysServices` L2+L3 = a multilayer switch, not a router.** The `sysServices L3→router` rule fired even when the L2 bit was *also* set, so an Arista/Catalyst **L3 switch** (which bridges *and* routes) was typed `router`. The rule now requires L2 to be absent: **pure L3 → router**, **L2+L3 → switch** — a general OSI-layer rule, no vendor names. (A rare Arista/Cisco device that is genuinely L3-only still classifies as a router.)
  - **(6) A WLAN controller is a `wlanctrl`, not a switch.** A Cisco vWLC reports `sysDescr` "Cisco Controller" with only the L2 bit, and both the generic path and the Cisco sysObjectID plugin fell through to `switch`. A vendor-neutral WLC signal (matched on the device's own descr/model: "wireless LAN controller", "mobility controller", AireOS, AIR-CT, Catalyst 9800, "Cisco Controller") now classifies as `wlanctrl`, and the Cisco sysObjectID plugin — which already resolves ASA→firewall, Aironet→AP, Catalyst→switch, ISR→router — gained the missing controller→`wlanctrl` case. The `wlanctrl` device type already existed; discovery now emits it.
  - **(7) Cisco "IOS" is not Apple "iOS".** The OS fingerprint's bare `ios` token was matching Cisco IOS / IOS-XE, Arista EOS, JunOS and RouterOS — so a network device with a weak signal could fingerprint as a *phone*. The match is now guarded against network-OS keywords.
  - Two vendors gained proper **sysObjectID recognition plugins**: a new `plugins/arista.js` (PEN 30065; infers switch/router from the *measured* sysServices, like `plugins/cisco.js`) so Arista is now confidently a switch instead of a low-confidence guess, and the completed `plugins/cisco.js` above. These are the same sysObjectID-recognition layer used for ~20 other vendors — not OUI/vendor-default guesses. An SNMP class confirmed by matching sysServices is now high-confidence on its own merit (it no longer piggybacked on the incidental iOS mismatch).
- Mirrored in the legacy classifier so the legacy↔fusion parity holds; validated end-to-end by live scans of both the real home LAN `192.168.1.0/24` (`.100` router→**switch**, `.128` Shield pc→**tv**, `.251` Chromecast iot→**tv**, `.218` Huawei tablet pc→**mobile**, two smart TVs correctly typed) and the multivendor lab `10.10.99.0/24` (Arista→**switch** @99, Cisco vWLC switch→**wlanctrl** @99, Cisco IOS switch no longer flagged `mobile`), with **every** already-correct device unchanged. `engine/fusion-scorer.js`, `server/classify.js`, `plugins/os-fingerprint.js`, `plugins/cisco.js`, `plugins/arista.js` (new), `server/netscan.js`, `server/routes/discovery.js` — **no `plugins/oui/*` vendor file touched**; +12 unit tests (G1–G7, Cast, mobile, WLC vendor-neutral) and 7 new legacy↔fusion parity cases. The `mobile` and `wlanctrl` device types already existed for manual placement; this is the first time discovery emits them.

## 2026-07-07 — Discovery trusts one classifier + a Windows host is no longer mistyped by its NIC vendor

### Changed
- **The Discover results now defer to the (authoritative) server classifier, and a file-sharing Windows host is typed as a computer even when its MAC vendor says "printer".** Three gaps, surfaced by a live scan: **(1)** discovery kept **two** classifiers — the strong server-side fusion engine (sysObjectID + OUI + product regexes) and a weaker client-side `_guessType` — and the client's copy took over whenever the server's confidence was merely low, so a device the server had already typed correctly could be relabeled by the thinner client heuristic. The client now **trusts the server class whenever the server found a real signal** (SNMP/sysObjectID, high aggregate score, *or* a fusion confidence above the baseline-fallback floor); the client `_guessType`/`_discSanitizeDeviceClass` stay only as a genuine-gap safety net for devices the server couldn't type at all (manual-first refinement, e.g. the AP branch). One classifier of record, far less divergence. **(2)** the deep-scan already ran NetBIOS (`nbtstat`) but **threw away the role information** — it now carries the suffix codes into classification (`<1B>/<1C>` Domain Controller → server; `<20>` Server-service → server unless a stronger nas/printer signal wins; otherwise a responding host → pc). *Honest caveat, confirmed live: NetBIOS-over-TCP/IP is off by default on modern Windows — nbtstat returns nothing on current PCs — so this mainly helps legacy/domain networks.* **(3)** the signal that actually works on modern Windows is **SMB**: a host with port 445 open and **enumerated file shares** (or RDP/WSD) and **no print ports** (9100/515/631) is a *computer*, not a printer — so it now scores as `pc`, beating the OUI-vendor inference (a Canon/HP NIC would otherwise drag a real Windows PC to "printer"). A device that *does* expose print ports stays a printer, and a NAS keeps its stronger `nas` signal. Validated live on the real LAN: a Windows PC with a printer-vendor NIC + 9 SMB shares flips from **printer → pc**, while the NAS on the same subnet stays a NAS. Mirrored in the legacy classifier for parity. `src/app-discovery.js` (consolidation gate), `server/routes/discovery.js` (propagate NetBIOS role flags), `engine/fusion-scorer.js`, `server/classify.js` (+2 unit tests). Builds on the vendor≠type guardrail below.

## 2026-07-07 — Device type is no longer decided by the vendor's company name (classification precision)

### Fixed
- **Discovery mis-typed devices by words inside their vendor's company name.** After vendor resolution switched to the full IANA Private Enterprise Numbers registry (~66k uncurated company names), the resolved name was folded into the text the fusion classifier keyword-matches for type — so a **"Gateway Inc."** PC (a real PC maker) became a **router** (confidence 86), an org literally named **"SWITCH"** became a **switch** (84), and **"Data Switch Corp"** / **"Firewall Services"** were mistyped the same way — worse, at *high* confidence. Researched how the field does it (nmap, Fingerbank, netdisco/SNMP::Info, runZero, ManageEngine): **unanimously, vendor identity and device type are kept as separate axes and nobody derives type by keyword-matching the company name** — nmap even classifies "Trapeze Networks" as a WAP, not a network device. Three changes, informed by that consensus and validated by a before/after diff over a 20-device corpus (only the 5 polluted cases changed; every legitimate case — MikroTik→router, Catalyst→switch, Fortinet→firewall, Synology→NAS, HP→printer, Hikvision→webcam, APC→UPS, Aruba→AP, PC/server/hypervisor — unchanged): **(1)** the generic type-nouns `gateway|switch|router|firewall` are **stripped from the vendor string before it enters type-matching** — the vendor's real *brand* tokens (mikrotik, fortinet, catalyst…) still vote, and the full name still shows in the vendor column (vendor = identity, never a type keyword). **(2)** the low-confidence fallback now reads the measured **sysServices** OSI-layer bits (L4+L7→host/server, L2→switch, L3→router) instead of blindly guessing "switch". **(3)** confidence gets a **contradiction discount** (Fingerbank-style): when a competing type scores close to the winner, confidence drops — honest uncertainty the user can review (manual-first), instead of a confident wrong label. The guardrail is mirrored in the legacy classifier so the legacy↔fusion parity holds. `engine/fusion-scorer.js`, `server/classify.js` (+4 unit tests: vendor-noun guardrail, brand preserved, measured fallback, contradiction discount). This is a pre-existing regression (commit `948ad0e`), independent of the SNMP-sync/topology work.

## 2026-07-07 — Presence audit no longer greys a whole subnet it never observed (multi-fabric)

## 2026-07-07 — Sync no longer stalls on a single slow/half-answering SNMP device

### Fixed
- **One unresponsive SNMP device could make a whole-project Sync appear to hang and never rebuild the topology.** Reproduced live on a mixed real-LAN + lab project: a firewall that answered `sysName` but then **timed out on every table walk** (ifDescr, LLDP, the bridge/FDB tables, ARP) burned **~70 s by itself** — each subtree paid its full 5 s timeout (plus a retry on the FDB group), and because the Sync's topology-enrichment phase queried devices **one at a time**, that 70 s was added *in series* to the Sync. With no client-side request timeout and no `try/finally` around the auto-link step, a device that never returned could also leave the Sync flag stuck (`_snmpSyncing`), freezing the button on "Topology…". Three vendor-neutral fixes: **(1)** the SNMP walk now honours a **wall-clock deadline** — once the budget is spent it stops and returns the partial data already gathered (topology is an *enrichment*, manual-first: a slow device yields fewer links, never invented ones). The budget derives from the per-device timeout (`3×`, floor 15 s for the neighbour walk; `5×`, floor 25 s for the inventory poll) so it scales if you raise the timeout for genuinely slow-but-real gear, and a late timeout no longer pays for a retry. Override with `SNMP_WALK_DEADLINE_MS`. Measured on the real device: **70.7 s → ~18 s**. **(2)** The Sync's topology phase now issues the `/api/topology` calls **in parallel batches of 5** (like the poll phase) with an **AbortController** client-side timeout, so one slow device no longer serialises the whole reconstruction and a hung endpoint can't stick the Sync forever; result processing stays sequential and deterministic (golden unchanged). **(3)** `pollAllSNMP` now runs inside `try/finally`, so the sync flag, Save button and Topology button are **always** reset even on an unexpected error. `drivers/snmp.js` (+4 unit tests: deadline skips remaining subtrees, no-retry past deadline, budget derivation, default = no cap), `src/app-autolink.js`, `src/app-snmp.js`.

## 2026-07-07 — Optional stealth (anti-IDS) pacing for the base subnet sweep

### Added
- **A stealth mode for the subnet scan that stays under an IDS's radar.** The base host-discovery sweep (pinging a whole subnet of *unknown* IPs) is the one phase with a scan signature: fired fast and in parallel it can trip a rate-based IDS/IPS on the very network you're documenting. Stealth mode (**opt-in**, `POST /api/discover` with `stealth: true` or `scanDelay: <ms>`) **serialises the sweep** (concurrency 1) and **spaces each probe by a jittered delay** (default 400 ms, ±30% — a *fixed* interval is itself a detectable cadence, so the jitter breaks it), the "polite"/T2 profile nmap uses to avoid tripping detectors. It applies **only** to the base sweep of unknown IPs; the deep/LLDP-CDP polling of already-discovered, authenticated SNMP devices stays parallel (`CRAWL_POOL`) since targeted polling of known hosts is not a scan signature. Default is unchanged (fast, parallel). No hosts are lost — validated live: same alive hosts found with stealth on vs off, only slower. New pure `_stealthDelayMs(base, jitterPct, rand)` in `server/netscan.js` (+4 unit tests); `server/routes/discovery.js`. Exposed in the Discover modal as a 3-level **Cadence** selector (Normal / Safe / Stealth) replacing the old "safe mode" checkbox — verified live in-browser that each level sends the right flags (`netmapper.html`, `src/app-discovery.js`, i18n it/en).

## 2026-07-07 — Topology crawl polls devices in parallel (IDS-aware), same result

### Changed
- **The LLDP/CDP topology crawl polled one device at a time**, so a single slow or unreachable device stalled the whole crawl and the wall-clock was the *sum* of every device's probe+poll time. The neighbour-expansion phase now runs as a **level-synchronised BFS with a bounded worker pool** (`server/crawl-bfs.js`, extracted from the SSE route and unit-tested): devices at the same BFS depth are probed/polled concurrently, then a barrier processes the results **in deterministic IP order** — so dedup-by-`sysName` and first-discoverer wins are stable and `pool=1` and `pool=N` produce **identical output** (proven by a unit test with skewed latencies and live on the lab: same device set at pool 1 vs 4). This is the *deep* phase only — polling of already-discovered, authenticated SNMP agents, which is not a scan signature; the base host-discovery sweep is unchanged and stays sequential/paced (**anti-IDS**). The pool is **`CRAWL_POOL` (default 4, 1-32)** — deliberately low so the socket footprint equals the pool (Raspberry-friendly) and because lab measurements show the returns knee at ~4-6 (the floor is the slowest single device). Measured ~1.3-4x faster depending on device count and how many are slow/dead. `server/crawl-bfs.js` (new, +11 tests), `server/routes/discovery.js`.

## 2026-07-07 — Topology overlay render batched (no layout thrashing), no output change

### Changed
- **Drawing the topology overlay caused heavy layout thrashing on large networks.** Each cable/fan-out line did two `document.querySelector` calls (O(DOM) each) and read a tile's `offsetWidth`/`offsetHeight` (a forced synchronous reflow) *while appending SVG nodes to the live overlay* — so every read after an append triggered a full reflow (the classic read→write→read anti-pattern). The render now (1) builds the SVG in a **DocumentFragment** and appends it **once** at the end — the live DOM isn't touched during the loop, so the `offsetWidth` reads no longer force a reflow per line — and (2) builds an **O(1) element index** (id → tile) once per render instead of two `querySelector` per line. Same SVG output by construction (identical elements, order and attributes; golden render + full suite green). Micro-benchmark of the exact pattern (1200 lines): **724 ms → 7 ms (~100x)**; combined with the earlier `buildTopoLines` speedup the whole topology view is dramatically faster on big maps. `src/app-topology-overlay.js`, `src/app-popup.js`.

Manual‑first: it points and explains, it never applies anything (`lib/onboarding.js`, pure & tested).
  - **Help grounded on the real UI** — for *"how do I X in InfraNet"* questions the assistant is grounded on the **actual command surface** — the buttons and their tooltips, **derived** from `netmapper.html` + the i18n labels (`lib/ui-catalog.js`), plus a short curated cheat‑sheet of the key flows — so it cites the **real button label** (e.g. *"click Discover"*) and doesn't invent menus. The catalog is generated from the single‑source‑of‑truth UI, so it never drifts from the app.

- **Visible lock for documented values** — a clickable lock (🔒) next to **IP**, **hostname** and a **port's VLAN** in the Properties panel. It is **not a new mechanism**: it surfaces the manual‑first protection that already existed. Locked = the value is your decision (the Sync leaves it untouched and *Verify documentation* flags any divergence from the network); unlocked = the field follows the network again. Reuses the existing `ipManual` / `hostnameManual` / `vlanOvr` pins — no engine change.

## 2026-06-28

### Added
- **IPAM occupancy per VLAN** — the VLAN's IPAM card now shows **real address usage**: capacity for the declared subnet, a usage bar (amber near full) and a breakdown of **documented vs DHCP‑only vs free** addresses. Occupancy combines documented IPs with active DHCP leases ("reality on the wire"). Pure, tested engine (`lib/ipam.js`).
- **Adopt from leases** — a per‑VLAN "**N undocumented → Adopt**" shortcut on the IPAM card maps DHCP‑only leases straight into the Adopt picker (MAC + IP + hostname), so the adopted device is born documented — no full Reality Check required.
- **"Management VLAN" role** — mark a VLAN as management (anti‑guest). An unknown device on a management VLAN is treated as **infrastructure** (never collapsed as BYOD) and flagged with a red **"On management VLAN"** security badge (possible intruder); Adopt proposes it as infra.
- **Import a discovered device as a host VM** — drag a floor tile (e.g. a VM that showed up as a loose PC) onto a host's **"Import VM"** drop‑zone in its Properties to absorb it into `node.vms[]`, inheriting name/IP/MAC. Its MAC joins the Drift known set, so it stops being reported as undocumented. The import fires **only** when the tile is released inside the drop‑zone; dropping anywhere else just repositions it (or snaps it back if released on a panel). Undoable with Ctrl+Z.
- **DHCP lease import (cross‑VLAN reality)** — paste or load a DHCP lease table (auto‑detects ISC dhcpd / dnsmasq / Kea CSV / generic CSV; covers pfSense / OPNsense / MikroTik / Synology / Windows exports) to feed the documentation check with authoritative MAC ↔ IP across **all VLANs** — the piece local ARP can't see behind an L3 firewall. Multiple servers accumulate as persisted sources (merged by MAC, freshest lease wins). A lease table is treated as an **identity map, not a liveness probe** (a documented device missing from leases is *unverifiable*, never wrongly *absent*; expired leases ignored; manually‑pinned IPs are never silently overwritten). Pure parser (`lib/dhcp-lease.js`); the file/paste path is built‑in, live vendor pull is a separate optional driver pack.
- **Endpoint / BYOD transparency** — undocumented entries that look like user devices (guest VLAN, crowded uplink port, or randomized "private" MAC) are collapsed into a *user/BYOD* group so the actionable infrastructure stays clean; each hidden row explains **why** in plain language (technical signal in the tooltip), with a toggle to reveal them inline.

### Changed
- **Uniform floor ↔ rack interaction** — floor devices now behave like rack devices: **single click selects**, **double click opens Properties**. (Previously a single click on a floor device opened its Properties.)

### Fixed
- Double‑click on a floor device now reliably opens its Properties — the native `dblclick` never fires on floor nodes because the floor re‑renders between the two clicks, so it's now detected manually by timestamp (same scheme already used for rack devices and ports).
- VM import no longer triggers by accident: the decision is taken from the **actual release point**, not from the last drag‑move, so a fast drag that passes over the drop‑zone but is released elsewhere no longer absorbs the device.
